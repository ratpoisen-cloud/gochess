# Проект: GoChess
**Описание:** Веб-приложение для игры в шахматы с друзьями в реальном времени.

## 🛠 Технологический стек
- **Frontend:** React (Vite)
- **UI/Стилизация:** Tailwind CSS
- **Шахматная логика:** `chess.js` (валидация ходов, состояние партии)
- **Отображение доски:** `react-chessboard` (или аналог)
- **Backend / База данных:** Supabase (PostgreSQL, Auth, Realtime)

## 📐 Архитектура и База Данных (Supabase)
Основные таблицы, с которыми мы работаем:
1. `profiles`: профили пользователей (связано с Supabase Auth).
2. `games`: активные и завершенные партии (id, white_player_id, black_player_id, status, fen_state).
3. `moves`: история ходов для конкретной партии.

*Синхронизация:* Используем Supabase Realtime для мгновенной передачи ходов и реакций (графические PNG-эмодзи).

## 🤖 Правила для ИИ-ассистента (Vibecoding Rules)

При написании кода для этого проекта строго соблюдай следующие правила:

1. **Графические Эмодзи:** Всегда используй ассеты из `public/emojis/board/` для реакций на доске. Константы в `src/lib/emojis.ts`.
2. **Фокус на чистоте:** Пиши модульный, читаемый код. Разделяй логику шахмат (движок), интерфейс (доску) и работу с сетью (Supabase).
2. **Шаги (Step-by-Step):** Не пытайся написать всё приложение за один ответ. Решай задачу по частям, жди подтверждения работоспособности.
3. **Безопасность (RLS):** При написании SQL-миграций для Supabase всегда учитывай Row Level Security (RLS). Пользователи должны изменять только свои партии.
4. **Обработка ошибок:** Всегда оборачивай сетевые запросы к Supabase в `try/catch` и выводи понятные сообщения об ошибках.
5. **Состояние игры:** Всегда опирайся на FEN-строку (Forsyth–Edwards Notation) как на единственный источник истины о состоянии доски.
6. **Никаких заглушек ради заглушек:** Если пишешь функцию, пиши её рабочую версию. Если нужен сторонний пакет, укажи команду для его установки.
